# 隐私与 AI 边界

XplorOne 被设计为一款本地优先的桌面财务工作台。

这份文档用于说明 XplorOne 如何处理本地数据、AI 参与方式、模型 API Key、本地 API token、MCP token 以及写入边界。

它是一份产品边界说明，不是法律意义上的隐私政策。

## 1. 默认本地优先

XplorOne 当前正式发布线是桌面应用。

核心桌面工作流围绕本地应用数据展开，包括：

- 账本
- 账户
- 分类
- 流水
- 报表
- 预算
- 设置与本地配置元信息

核心财务记录保存在用户自己的本地应用数据环境中。

XplorOne 并不是那种要求你一开始就把整个财务工作流上传到云端系统里的产品。

## 2. 没有 API Key 也能使用什么

你不需要模型 API Key，也可以把 XplorOne 作为本地优先财务工作台使用。

即使没有 API Key，你也可以使用这些核心本地能力：

- 本地记账
- 记一笔
- 账本、账户、分类和流水
- 报表与财务视图
- 基础本地查询
- 当前版本支持的备份、恢复、导出、导入与归档工作流

模型 API Key 只在你需要 AI 辅助能力时才需要。

## 3. 哪些能力需要 API Key

只有当你需要 AI 辅助能力时，才需要自行配置模型 API Key。

根据当前启用的工作流，AI 辅助能力可能包括：

- AI 助手中的分析与解释
- 更开放的财务相关对话
- 明确进入 AI 工作流时的模型辅助理解
- 启用模型参与时的录入草稿辅助

API Key 的作用，是把 XplorOne 连接到用户自己选择的模型服务。

普通本地记账、报表、财务视图和基础本地查询并不依赖 API Key。

## 4. 模型 API Key 保存方式

模型 API Key 会被视为敏感的本地凭据数据。

XplorOne 会通过 Electron `safeStorage` 或等效的系统级本地凭据保护机制保存模型凭据。

模型 API Key 不应以明文形式写入账本数据库。

非敏感的模型配置元信息可以保存在本地，例如：

- provider 名称
- base URL
- model name
- 启用状态
- 配置状态

这些元信息用于帮助 XplorOne 记住当前使用哪组模型配置，但不会把模型密钥当成普通账本数据处理。

用户可以在应用设置中移除、替换或重新配置自己的模型 API Key。

## 5. 本地 API 与 MCP token 边界

XplorOne 可能会为本地集成路径使用本地 token。

这些 token 和模型 API Key 不是同一类东西。

- **模型 API Key** 用于连接用户选择的大模型服务。
- **本地 API token** 用于保护 XplorOne 本地 HTTP API 的访问。
- **MCP client token** 用于授权 Codex、WorkBuddy 这类外部 agent 访问 XplorOne 的本地只读 MCP / query 路径。

本地 API 与 MCP 访问主要面向本机回环环境中的本地集成。

它们不应该被当作公开 Web API 凭据使用。

MCP client token 会被视为敏感的本地凭据数据，并通过 Electron `safeStorage` 或等效的系统级保护机制保存。

用户不应：

- 分享这些 token
- 把 token 粘贴到公开 issue
- 把 token 提交到 GitHub
- 把 token 发送给第三方
- 在截图或公开支持信息里暴露 token

如有需要，本地集成 token 应可以在应用设置中移除、重置或重新生成。

## 6. 哪些数据可能发送给模型服务商

核心本地功能不需要向模型服务商发送数据。

当你使用 AI 辅助能力时，XplorOne 可能会把完成该 AI 任务所需的相关信息发送给用户自行配置的模型服务。

根据具体工作流，这些信息可能包括：

- 用户当前的问题或指令
- 当前请求所需的结构化财务上下文
- 选定的报表或查询上下文
- 录入草稿上下文
- 最近对话上下文
- AI 个性化摘要，仅在用户启用 AI 个性化时使用

XplorOne 的目标是发送与任务相关的上下文，而不是默认把整个本地数据库发送给模型服务。

用户应选择自己信任的模型服务商，并了解该服务商自己的数据与隐私条款。

## 7. 本地助手与 AI 助手边界

XplorOne 不把所有助手交互都当成同一种能力。

### 本地助手查询

在当前对话结构中，本地助手的查询由本地查询内核处理。

它可以通过本地规则回答、发起补问，或返回不支持边界，而不是悄悄 fallback 到 AI 猜 intent。

受支持的本地查询不需要模型 API Key。

### 本地助手录入

录入工作流以草稿为核心。

录入工作流会先准备草稿，草稿必须经过用户检查和确认后，才会真正写入。

如果某个工作流启用了模型参与，它也仍然属于明确的助手流程，而不是自动写入路径。

### AI 助手分析

AI 助手分析可能会使用结构化财务上下文，生成解释、趋势和建议。

它的目标是帮助用户理解财务数据，而不是替代用户自己的判断。

### AI 助手自由对话

AI 助手自由对话用于更开放的财务相关交流。

它不等同于直接账本查询。
在当前发布线中，自由对话并不是默认直接访问账本的路径。

## 8. AI 写入边界

AI 不会自动写入账本。

涉及写入的操作必须由用户确认。

这尤其适用于：

- 创建流水
- 提交录入草稿
- 任何会改变账本数据的工作流

XplorOne 的设计目标是：本地助手可以处理受支持的本地查询和录入工作流，AI 助手可以辅助分析和更开放的财务相关对话。

是否写入仍然由用户确认。

## 9. 本地 API 与 MCP 只读边界

XplorOne 包含面向 AI-native 工作流的本地集成能力。

当前 MCP / query 路径被设计为 token 保护的本地只读访问路径。

当前边界包括：

- MCP 访问是只读的
- MCP 不暴露写入操作
- MCP 查询会经过 XplorOne 受控的本地 API 与查询链路
- MCP 不直接读取数据库
- MCP 围绕当前激活账本作用域设计
- 外部 agent 接收结构化查询结果，并由其自身模型负责解释结果

这使得外部 agent 可以查询相关财务数据，同时保持 XplorOne 的应用边界清晰。

## 10. 备份、导出与归档边界

XplorOne 根据当前发布状态，提供备份、恢复、导出、导入和归档等工作流。

一般边界是：

- 账本导出和归档包用于保存财务数据，不应暴露明文凭据。
- 模型 API Key 与 MCP token 不应以明文形式出现在导出的账本文件中。
- full-app 备份工作流可能包含受保护的凭据元数据，用于当前机器环境下的恢复。
- 受保护凭据备份不等于可跨机器移植的明文凭据导出。
- 即使凭据没有明文暴露，财务数据本身依然敏感，用户仍应妥善保存备份文件。

如果你在不同机器之间迁移数据，可能需要根据备份类型和操作系统凭据环境，重新配置模型 API Key 或本地集成 token。

## 11. 用户需要注意什么

XplorOne 是本地优先产品，但本地优先并不意味着用户不需要承担任何安全责任。

用户应当：

- 保护自己的电脑账户
- 妥善保存备份和导出文件
- 不分享模型 API Key 或本地集成 token
- 使用自己信任的模型服务商
- 了解模型服务商自己的数据政策
- 不在公开 issue 中粘贴敏感财务数据
- 如果 token 可能泄露，应及时重置本地 token

如果你提交 bug，请在发布前删除私人财务数据、API Key、token 和个人信息。

## 12. XplorOne 不是什么

XplorOne 不是：

- 云端优先的会计套件
- 默认把所有数据上传到云端的托管式 AI 财务服务
- 面向所有团队、所有地区的企业级财务系统
- 允许 AI 不经确认就自动改写账本的系统
- 公开 Web API 服务

更准确地说，XplorOne 是一款本地优先的桌面财务工作台，并提供可选的 AI 辅助能力和受控的本地集成路径。
