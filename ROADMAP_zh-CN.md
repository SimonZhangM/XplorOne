# XplorOne 路线图

这份路线图用于说明 XplorOne 当前的产品方向。

它是方向性说明，不是交付承诺。
后续优先级可能会根据产品稳定性、用户反馈、技术约束和发布准备情况调整。

XplorOne 当前的核心方向，是成为一款可靠的本地优先桌面财务工作台，服务于超级个体、自由职业者、一人公司经营者、顾问、独立开发者和小型工作室主理人。

## 产品原则

XplorOne 的路线图遵循几个基本原则：

- **默认本地优先**
  核心财务记录应尽量留在用户自己的控制范围内。

- **桌面工作流优先**
  当前正式发布线聚焦桌面应用。

- **AI 是可选增强层**
  即使没有模型 API Key，XplorOne 也应该可以完成核心本地记账、报表和基础查询。AI 用于增强查询、录入草稿和分析，而不是取代核心本地工作流。

- **写入边界必须清楚**
  AI 可以辅助理解、分析和准备草稿，但涉及写入的操作必须由用户确认。

- **数据与集成边界要明确**
  本地 API 与 MCP 集成应保持受控、token 保护和边界清晰。

- **优先解决真实财务工作流，而不是功能膨胀**
  XplorOne 的目标不是成为一个大而全的企业级会计套件，而是帮助独立经营者理解和管理自己的财务现实。

## Now｜当前阶段

当前阶段重点是稳定第一版公开发布基础。

### Windows 桌面发布稳定性

当前重点包括：

- 提升 Windows 桌面版发布可靠性
- 让安装与更新体验更稳定
- 保持清晰的版本说明
- 减少安装和启动过程中的摩擦
- 确保发布产物不包含本地私有运行数据、凭据、token 或本地数据库

### GitHub 产品公开入口

当前重点包括：

- README 与双语文档
- 快速开始
- 功能导览
- 隐私与 AI 边界
- 当前限制说明
- BYOK 配置说明
- FAQ
- 支持说明与 Issue 模板
- Changelog 与面向用户的软件版本历史

目标是让 GitHub 仓库成为清楚的公开产品入口，承接版本发布、文档说明、路线更新和社区反馈。

### 核心本地财务工作流

当前重点是继续完善日常使用路径：

- 记一笔
- 首页工作台
- 流水查看
- 账户与分类
- 报表与财务视图
- 备份、恢复、导出、导入和归档工作流

第一阶段的产品目标很简单：

> 用户应该可以不依赖 AI，就完成财务记录、查看账本，并理解当前财务状态。

### AI 辅助工作流

当前重点包括：

- 自然语言查询
- 录入草稿准备
- 分析与解释
- 双语 AI 表现
- 模型配置清晰度
- 回复语言行为
- 涉及写入工作流的确认机制

AI 仍然是可选能力。
核心本地记账、报表与基础查询应继续在没有模型 API Key 的情况下可用。

## Next｜下一阶段

下一阶段重点是让 XplorOne 更适合处理真实世界中的财务数据，并进一步增强本地优先的 AI 财务工作台能力。

### 银行账单与主流支付工具导入

这是 Next 阶段的重点方向。

XplorOne 需要增强对真实银行账单和主流支付工具导出文件的导入支持。

重点包括：

- 银行账单导入
- 主流支付工具导入
- 更好的 CSV 与表格文件导入
- 格式识别与字段映射
- 导入预览
- 重复数据识别
- 最终导入前的用户确认
- 更安全的导入后流水检查
- 更清楚的导入错误提示

当前目标不是立刻宣称支持所有银行、所有支付平台或所有外部格式。

更现实的目标是：先建立可靠的导入基础，让用户可以逐步把历史记录转入结构化的本地账本。

### AI 查询与录入草稿增强

后续 AI 能力可能包括：

- 更好的本地优先查询路由
- 更强的自然语言理解
- 更准确的时间范围识别
- 更好的分类和账户匹配
- 更好的录入草稿修正
- 更清晰的缺失字段追问
- 更稳定的双语查询体验
- 更有用的财务结果解释

写入边界保持不变：AI 生成的写入动作必须经过用户确认。

### 报表与财务视图

后续报表改进可能包括：

- 更清晰的现金流视图
- 更强的收支分析
- 更好的资产负债可见性
- 更好的图表解释
- 从报表跳转到底层流水的体验优化
- 更一致的中英文标签与术语

目标是帮助用户理解财务模式，而不只是看到一组数字。

### 数据控制与可迁移性

后续工作可能包括：

- 更清晰的备份与恢复说明
- 更好的导出选项
- 更稳定的归档工作流
- 更安全的导入校验
- 更清楚的数据迁移引导
- 更好地处理跨机器迁移时的凭据边界

XplorOne 应继续区分财务数据与凭据数据，不把它们混为一谈。

### BYOK 与模型服务体验

后续工作可能包括：

- 更清楚的模型服务配置引导
- 更稳定的连接测试
- 更好的错误提示
- 更好的服务商兼容处理
- 更稳定的回复语言行为
- 更清楚地区分模型 API Key 与本地集成 token

核心产品仍应在没有模型 API Key 的情况下可用。

### 本地 MCP 与外部 agent 工作流

后续工作可能包括：

- 完善本地只读 MCP 工作流
- 完善 Codex 与 WorkBuddy 使用文档
- 更清晰的 MCP 状态与 token 管理
- 更好的结构化结果格式
- 更多面向外部 agent 的查询示例
- 更清楚地区分桌面端用户工作流与外部 agent 工作流

当前 MCP 方向仍保持只读。

## Later｜后续阶段

当核心工作流更加稳定后，后续阶段可能继续探索更广的产品扩展。

### macOS 与更多平台评估

XplorOne 后续可能评估更多平台支持。

但当前口径仍然是：

- Windows 是当前正式发布线
- macOS、Linux、移动端或浏览器正式用户版本，除非明确发布，否则不应默认为可用
- 平台扩展不应牺牲核心产品稳定性

### 更广的导入覆盖

当导入基础稳定后，XplorOne 可以逐步扩大对更多账单类型、导出格式和支付工作流的支持。

### 更深入的财务分析

后续分析能力可能包括：

- 更强的期间对比
- 更好的项目维度财务视图
- 更好的成员相关分析
- 更深入的现金流解释
- 更有用的财务健康摘要
- 基于本地账本结构的更实用建议

这些能力仍应建立在用户可控数据和清晰 AI 边界之上。

### 更多 agent 与集成工作流

XplorOne 可能继续探索外部 agent 工作流、本地 skills 和受控集成路径。

产品方向仍然保持：

- 本地优先
- token 保护
- 边界明确
- 涉及写入的操作由用户确认

---

## 当前不作为核心方向

XplorOne 当前并不试图成为：

- 云端优先的会计套件
- 托管式多人会计平台
- 面向所有地区的完整企业财务系统
- 税务申报产品
- 法律或会计合规服务
- 必须依赖 AI 才能使用的产品
- 允许 AI 不经确认就自动改写账本的系统

这些是有意保留的产品边界，而不是暂时没补上的空白。

## 如何反馈路线图建议

欢迎对路线图提出反馈。

最有帮助的反馈通常会说明：

- 你想解决什么问题
- 你目前怎么处理这个工作流
- XplorOne 现在缺少什么
- 这是个人、自由职业、工作室还是业务经营场景
- 这个需求与记账、导入、报表、AI、MCP、备份还是设置有关

你可以使用：

- [Discussions](../../discussions) 反馈路线图想法和产品讨论
- [Issues](../../issues) 提交明确 bug 和可处理的问题
- [支持说明](./SUPPORT_zh-CN.md) 了解反馈方式

请不要在公开反馈中发布私人财务数据、API Key、本地 API token、MCP token、授权码或未脱敏截图。
